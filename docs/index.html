1임베딩이란? — 의미를 숫자 벡터로

임베딩(Embedding)은 텍스트·이미지·코드를 고정 길이 숫자 벡터로 변환하는 모델입니다. 의미가 비슷한 문장은 벡터 공간에서 가까이, 다른 의미는 멀리 위치합니다. RAG의 검색 레이어가 이걸 사용합니다.

입력출력 (예: bge-m3 · 1024차원)코사인 유사도
"POS 거래 이상 탐지"[0.023, -0.156, 0.781, ...] (1024개)—
"매장 결제 이상 패턴"[0.041, -0.143, 0.755, ...]0.91 (가까움)
"오늘 점심 메뉴"[-0.412, 0.587, 0.022, ...]0.12 (먼)

핵심 활용: 사내 문서 N개를 미리 임베딩 → 사용자 질의도 임베딩 → 가장 가까운 문서 K개 반환 (검색의 근본). LLM 위키·QMD·RAG가 다 이 패턴.

3청킹 전략 — 자료 유형별

임베딩의 입력 단위는 chunk(조각). 자료 유형별로 최적 청킹 방식이 다릅니다.

자료 유형청킹 전략이유
마크다운·기획서헤더 단위 (## 섹션) + 토큰 한도 (≈500토큰)의미 단위 보존
코드함수·클래스 단위 (AST 기반)로직 단위 보존 — 임의 분할 시 의미 깨짐
PDF (문서)페이지 단위 + 표는 별도 chunk레이아웃 보존 · 표 의미 별도 보존
DB (테이블)행 단위 + 컬럼 메타데이터관계형 의미 보존
긴 글 (논문·책)슬라이딩 윈도우 + 오버랩 (50~100토큰)경계 정보 손실 방지

겹침(overlap): 인접 chunk 끝-시작 50~100토큰 공유 → 경계 질의에서 답이 끊기는 문제 방지. 그러나 인덱싱 비용 1.1~1.3배 증가.

4임베딩 품질 검증

임베딩이 "잘 됐는지" 검증하는 표준 방법:

  1. R@K (Recall at K) — 정답 문서가 상위 K개 결과 안에 들어오는 비율. 보통 R@5 · R@10 측정.
  2. MRR (Mean Reciprocal Rank) — 정답이 몇 번째에 나왔는지의 역수 평균. 1위에 가까울수록 높음.
  3. NDCG (Normalized DCG) — 순위별 가중 평가. 상위에 좋은 결과가 있으면 점수 ↑.
  4. 골든셋(Goldset) 구축 — 사내 자료에서 질의-정답 쌍 50~100개 직접 작성 → 모델 비교 기준.

실 운영 팁: 한국어 자료 + 오픈 모델 조합은 영어 벤치마크 수치를 그대로 믿으면 안 됨. 본인 도메인 골든셋으로 직접 측정. bge-m3는 한국어 강함.

5모델 교체 비용 — 운영 시 가장 큰 함정

임베딩 모델은 한 번 정하면 전체 인덱스가 그 모델에 lock-in됩니다. 모델 변경 시:

  • 기존 벡터 DB 전체 재임베딩 필요 (300개 문서 = 작지만, 30만 개 = 큰 비용)
  • 벡터 차원이 다르면 DB 스키마 변경 (예: 768 → 1024)
  • 골든셋 재측정 필요 (성능 비교)
  • 임베딩 + chunk 시간 + 검수 = 보통 며칠~몇 주

실 운영 추천 | 초기 PoC는 가벼운 모델 (예: multilingual-e5-small · 384차원)로 시작 → 패턴 확정 후 본격 모델 (bge-m3 · 1024차원)로 마이그레이션. 한 번에 큰 모델 시작 시 운영 부담 ↑.

4운영 한계 — 실제로 부딪힌 문제

  • 검색 품질 — 한국어 자료는 영어 벤치마크처럼 안 됨. 골든셋 직접 측정 필수.
  • 환각 — GraphRAG가 관계를 잘못 추출하면 LLM이 그걸 사실로 인용. 그래프 lint 필요.
  • 유지보수 — 새 자료 추가 시 그래프 재구성 비용 ↑. 증분 업데이트 어려움.
  • LLM 호출 비용 — 그래프 빌드 = 엔티티·관계 추출 LLM 호출 다수. 대규모 코퍼스에서 부담.
  • 한국어 엔티티 추출 — 영어 모델은 한국어 엔티티를 자주 놓침. 사전·NER 모델 보정 필요.

3크론잡 — 자동 업데이트 + 임베딩

QMD를 무인 운영하는 핵심: 크론잡으로 정기 자동 인덱싱.

Linux · macOS · WSL — crontab
# 매시간 새 파일 자동 인덱싱
0 * * * * cd ~/notes && qmd update

# 매일 새벽 3시 전체 임베딩 재계산
0 3 * * * cd ~/notes && qmd embed --full

# 매주 일요일 인덱스 통계 로그
0 9 * * 0 qmd stats >> ~/qmd-weekly.log
Windows — 작업 스케줄러
# PowerShell — 매시간 작업 등록
$action = New-ScheduledTaskAction `
  -Execute "qmd" -Argument "update" `
  -WorkingDirectory "$env:USERPROFILE\notes"
$trigger = New-ScheduledTaskTrigger `
  -Once -At (Get-Date) `
  -RepetitionInterval (New-TimeSpan -Hours 1)
Register-ScheduledTask -TaskName "qmd-hourly-update" `
  -Action $action -Trigger $trigger

임베딩 조건 — 언제·무엇을

  • 매시간 — 새 파일 증분 인덱싱 (저비용 · 거의 즉시)
  • 매일 — 변경된 파일의 임베딩 재계산 (자료 수정분 반영)
  • 매주 — 전체 임베딩 재계산 (모델 업그레이드 후 또는 lint 후)
  • 이벤트 기반 — git push hook으로 즉시 인덱싱 (CI 통합)

5비LLM AI 전처리 — OCR · 텍스트 마이닝

마크다운화 전 단계 보조 도구. LLM 호출 없이 빠르게 전처리:

  • OCR — Tesseract(영문) · PaddleOCR(한국어 강) · Surya(레이아웃 보존) · Donut(end-to-end)
  • 표·차트 추출 — Camelot · Tabula · Microsoft Table Transformer
  • 수식 추출 — pix2tex · MathPix (LaTeX 변환)
  • 텍스트 마이닝 — KoNLPy(한국어 형태소) · spaCy · keyBERT(키워드) · BERTopic(토픽)
  • 감성 분석 — KoBERT · 한국어 sentiment model (huggingface)

조합 패턴 — 비LLM 전처리 + LLM 후처리:

[원본 PDF (스캔본)]
   ↓ PaddleOCR (비LLM)
[텍스트 + 레이아웃]
   ↓ Camelot (비LLM)
[표 데이터 별도 추출]
   ↓ LLM 후처리 (구조화 · 요약)
[깔끔한 마크다운] → 임베딩 → QMD

왜 비LLM 전처리? 토큰비 0 · 빠름 · 결정적 (deterministic). LLM은 마지막 요약·구조화 단계에만 쓰면 비용 효율 ↑. 회사 환경처럼 대량 자료 처리 시 필수 패턴.

Ouroboros (우로보로스)
Q00/ouroboros · Agent OS — 자기-진화 루프 · 한국어 README

"프롬프트하지 말고 명세하라" — 4단계 진화 루프로 모호한 요구사항을 정확한 명세로 변환.

단계역할
① InterviewSocratic Clarity — 모호함 ≤ 0.2까지 질문 반복. AI가 "이건 무슨 뜻인지" 묻고 사용자가 답하며 명세를 정련
② SeedOntological Precision — 인터뷰 결과로 근본 문제(symptom X · root cause) 정의. 명세의 seed 생성
③ Executeseed 기반 구현 — 코드 작성 · 시스템 구축. 명세가 정확할수록 산출물 정확
④ Evaluate실행 결과를 평가해 다음 사이클 명세에 피드백 — 진화 루프 형성. 매 사이클이 단순 반복이 아닌 진화

MCP 서버 자동 발견 (~/.ouroboros/mcp_servers.yaml). 본 강의 외 본인 프로젝트 설계 시 활용 추천.
